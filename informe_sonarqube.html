1. Introducción

1.1 ¿Qué es el análisis estático de código?

El análisis estático de código es una técnica de inspección del código fuente que se realiza sin ejecutar el programa. A diferencia de las pruebas dinámicas (que requieren correr la aplicación con entradas concretas), el análisis estático examina la estructura, la sintaxis y los patrones del código para detectar defectos potenciales antes de que el software llegue a producción.

En la práctica, un analizador estático recorre el árbol sintáctico abstracto (AST) del código fuente y lo compara contra un conjunto de reglas predefinidas que codifican buenas prácticas, patrones de error conocidos y estándares de seguridad. Cuando el código viola una regla, la herramienta reporta el hallazgo indicando el archivo, la línea y la severidad del problema.

1.2 Importancia en la verificación de software

Dentro del marco de Verificación y Validación (V&V), el análisis estático es una técnica de verificación: responde a la pregunta «¿estamos construyendo el producto correctamente?». Su importancia radica en tres aspectos clave:

  • Detección temprana: Los defectos encontrados durante el análisis estático cuestan entre 10 y 100 veces menos de corregir que los encontrados en producción, según el modelo de costo relativo de Boehm.
  • Cobertura exhaustiva: A diferencia de las pruebas dinámicas, el análisis estático examina todos los caminos posibles del código, incluyendo ramas que serían difíciles de alcanzar con pruebas.
  • Automatización continua: Las herramientas de análisis estático se integran fácilmente en pipelines de CI/CD, garantizando que cada commit pase por una revisión automática de calidad.

1.3 SonarQube como herramienta de análisis estático

SonarQube es una plataforma de código abierto (con versión Community gratuita) desarrollada por SonarSource que permite inspeccionar la calidad del código fuente de manera continua. Soporta más de 30 lenguajes de programación y organiza sus hallazgos en tres categorías principales:

CategoríaDescripciónEjemplos
Bug Error de lógica que probablemente causa comportamiento incorrecto en tiempo de ejecución Comparar Strings con == en Java, posibles NullPointerException
Vulnerability Debilidad de seguridad que podría ser explotada por un atacante Exposición de datos sensibles, algoritmos criptográficos débiles
Code Smell Indicador de problema de mantenibilidad que no es un error funcional pero dificulta la comprensión Nombres poco expresivos, números mágicos, métodos muy largos

SonarQube asigna a cada hallazgo una severidad (Blocker, Critical, Major, Minor, Info) y una estimación del tiempo de remediación, lo que permite priorizar las correcciones de forma objetiva.

4. Correcciones Realizadas

Corrección 1: Bug Critical — Comparación de Strings con ==
CampoDetalle
IssuesB1 (línea 12) y B2 (línea 32)
TipoBug
SeveridadCRITICAL
Archivosrc/main/java/ec/edu/epn/Pub.java
Commitfix: reemplazar comparaciones de String con == por .equals()

Descripción del problema

En Java, el operador == compara referencias de objetos, no el contenido de los strings. Cuando se compara drink == GT, Java verifica si las dos variables apuntan exactamente al mismo objeto en memoria, no si tienen el mismo valor textual.

En este código, las constantes (GT, BACARDI_SPECIAL, ONE_CIDER, etc.) son Strings literales definidos como static final. Cuando el valor de drink proviene de otra fuente (por ejemplo, un parámetro externo construido con new String("gt")), la comparación con == devuelve false aunque el contenido sea idéntico, causando que la lógica de validación y descuentos falle silenciosamente.

Código antes de la corrección

ANTES 
// Línea 12 — Bug B1: usa == para comparar Strings
if (amount > 2 && (drink == GT || drink == BACARDI_SPECIAL)) {

// Línea 32 — Bug B2: usa == para comparar Strings
if (student && (drink == ONE_CIDER || drink == ONE_BEER || drink == A_PROPER_CIDER)) {

Código después de la corrección

DESPUÉS 
// Línea 18 — Correcto: usa .equals() para comparar contenido
if (amount > MAX_COCKTAIL_AMOUNT && (drink.equals(GT) || drink.equals(BACARDI_SPECIAL))) {

// Línea 38 — Correcto: usa .equals() para comparar contenido

Corrección 2: Code Smell Major — Nombres de métodos sin significado
CampoDetalle
IssuesCS1 a CS6 (líneas 39–66)
TipoCode Smell
SeveridadMAJOR
Archivosrc/main/java/ec/edu/epn/Pub.java
Commitrefactor: renombrar métodos de ingredientes con nombres descriptivos

Descripción del problema

Los seis métodos privados (ingredient1() a ingredient6()) tienen nombres genéricos que no transmiten ninguna información sobre su propósito. Un lector del código no puede saber qué ingrediente representa cada método sin leer el comentario adjunto, lo que viola el principio de Clean Code de nombres que se expliquen a sí mismos.

SonarQube clasifica esto como Code Smell Major porque dificulta el mantenimiento: si se necesita cambiar el precio del ron, ¿cuál de los seis métodos hay que modificar? Con los comentarios eliminados (algo común en refactorizaciones apresuradas), el código se vuelve ininteligible.

Código antes de la corrección

ANTES 
//one unit of rum
private int ingredient1() { return 65; }

//one unit of grenadine
private int ingredient2() { return 10; }

//one unit of lime juice
private int ingredient3() { return 10; }

//one unit of green stuff
private int ingredient4() { return 10; }

//one unit of tonic water
private int ingredient5() { return 20; }

//one unit of gin
private int ingredient6() { return 85; }

Código después de la corrección

DESPUÉS 
private int rum()        { return RUM_PRICE; }
private int grenadine() { return GRENADINE_PRICE; }
private int limeJuice() { return LIME_JUICE_PRICE; }
private int greenStuff() { return GREEN_STUFF_PRICE; }
private int tonicWater() { return TONIC_WATER_PRICE; }
private int gin()        { return GIN_PRICE; }

Con los nuevos nombres, la fórmula del Gin & Tonic es inmediatamente comprensible:

Corrección 3: Code Smell Major — Números mágicos sin nombre
CampoDetalle
IssuesCS7 a CS12 (múltiples líneas)
TipoCode Smell
SeveridadMAJOR
Archivosrc/main/java/ec/edu/epn/Pub.java
Commitrefactor: reemplazar números mágicos por constantes nombradas

Descripción del problema

Un «número mágico» es un literal numérico en el código cuyo significado no es evidente sin contexto adicional. Por ejemplo, al leer price = 74, un mantenedor no sabe si 74 es un precio en céntimos, en coronas, un código de producto o un valor de configuración. Si el precio de la cerveza cambia, habría que buscar todas las ocurrencias del número 74 en el código, arriesgándose a modificar también usos no relacionados.

Código antes de la corrección

ANTES 
// Precios hardcodeados sin nombre — difícil de mantener
if (amount > 2 && ...) { ... }          // ¿Por qué 2?
if (drink.equals(ONE_BEER))  price = 74;  // ¿74 qué?
if (drink.equals(ONE_CIDER)) price = 103; // ¿103 qué?
if (drink.equals(A_PROPER_CIDER)) price = 110;
price = price - price/10;               // ¿El 10% de descuento?

Código después de la corrección

DESPUÉS 
// Constantes agrupadas al inicio de la clase — significado claro
private static final int BEER_PRICE            = 74;
private static final int CIDER_PRICE           = 103;
private static final int PROPER_CIDER_PRICE    = 110;
private static final int MAX_COCKTAIL_AMOUNT   = 2;
private static final int STUDENT_DISCOUNT_DIVISOR = 10;

private static final int RUM_PRICE        = 65;
private static final int GRENADINE_PRICE  = 10;
private static final int LIME_JUICE_PRICE = 10;
private static final int GREEN_STUFF_PRICE = 10;
private static final int TONIC_WATER_PRICE = 20;
private static final int GIN_PRICE        = 85;

price = price - price / STUDENT_DISCOUNT_DIVISOR;

4.1 Código final corregido — Pub.java

package ec.edu.epn;

public class Pub {
    public static final String ONE_BEER        = "hansa";
    public static final String ONE_CIDER       = "grans";
    public static final String A_PROPER_CIDER  = "strongbow";
    public static final String GT              = "gt";
    public static final String BACARDI_SPECIAL = "bacardi_special";

    private static final int BEER_PRICE             = 74;
    private static final int CIDER_PRICE            = 103;
    private static final int PROPER_CIDER_PRICE     = 110;
    private static final int MAX_COCKTAIL_AMOUNT    = 2;
    private static final int STUDENT_DISCOUNT_DIVISOR = 10;

    private static final int RUM_PRICE        = 65;
    private static final int GRENADINE_PRICE  = 10;
    private static final int LIME_JUICE_PRICE = 10;
    private static final int GREEN_STUFF_PRICE = 10;
    private static final int TONIC_WATER_PRICE = 20;
    private static final int GIN_PRICE        = 85;

    public int computeCost(String drink, boolean student, int amount) {
        if (amount > MAX_COCKTAIL_AMOUNT
                && (drink.equals(GT) || drink.equals(BACARDI_SPECIAL))) {
            throw new RuntimeException("Too many drinks, max 2.");
        }
        int price;
        if      (drink.equals(ONE_BEER))       price = BEER_PRICE;
        else if (drink.equals(ONE_CIDER))      price = CIDER_PRICE;
        else if (drink.equals(A_PROPER_CIDER)) price = PROPER_CIDER_PRICE;
        else if (drink.equals(GT))             price = gin() + tonicWater() + greenStuff();
        else if (drink.equals(BACARDI_SPECIAL)) price = gin()/2 + rum() + grenadine() + limeJuice();
        else throw new RuntimeException("No such drink exists");

        if (student && (drink.equals(ONE_CIDER) || drink.equals(ONE_BEER)
                || drink.equals(A_PROPER_CIDER))) {
            price = price - price / STUDENT_DISCOUNT_DIVISOR;
        }
        return price * amount;
    }

    private int rum()        { return RUM_PRICE; }
    private int grenadine() { return GRENADINE_PRICE; }
    private int limeJuice() { return LIME_JUICE_PRICE; }
    private int greenStuff() { return GREEN_STUFF_PRICE; }
    private int tonicWater() { return TONIC_WATER_PRICE; }
    private int gin()        { return GIN_PRICE; }
}

6. Conclusiones

Sobre la experiencia con SonarQube: El uso de una herramienta de análisis estático cambia fundamentalmente la forma en que se revisa el código. En lugar de depender de revisiones manuales que pueden omitir defectos sutiles, SonarQube ofrece una revisión sistemática y repetible en segundos. Lo más revelador fue comprobar que los dos bugs críticos (comparación con ==) no habrían sido detectados necesariamente por las pruebas unitarias existentes, porque las constantes de tipo String literal en Java se internan en el pool de strings y la comparación con == puede devolver true aunque sea semánticamente incorrecta. SonarQube detectó el problema de forma determinista sin importar el valor en tiempo de ejecución.

Relación con los principios de Clean Code: Las correcciones realizadas ilustran directamente varios principios del libro de Robert C. Martin:

  • «Use Intention-Revealing Names» (Cap. 2): renombrar ingredient1() a rum() hace que el código comunique su intención sin necesidad de comentarios.
  • «Avoid Magic Numbers» (Cap. 17): reemplazar 74 por BEER_PRICE convierte el código en documentación ejecutable; cualquier desarrollador sabe qué representa ese valor.
  • «Don't use == for String comparison»: aunque parece un error de principiante, es precisamente el tipo de defecto que se filtra en código bajo presión de tiempo y que solo una herramienta automatizada detecta de forma confiable.

Lecciones aprendidas: La integración de SonarQube en el flujo de trabajo debe hacerse desde el inicio del proyecto, no al final. Si se ejecuta el análisis por primera vez con cientos de issues acumulados, la deuda técnica es abrumadora. En cambio, si se configura como Quality Gate en el pipeline de CI/CD, cada commit pasa por la revisión automáticamente y los defectos se corrigen cuando el contexto todavía está fresco. Esta práctica, conocida como «shift-left testing», es uno de los pilares de la verificación continua de software.

Ventajas de SonarQube:

  • Detecta clases enteras de defectos (como comparaciones con ==) de forma exhaustiva en todo el código base.
  • Proporciona estimaciones de tiempo de remediación y tendencias históricas, útiles para gestión de proyectos.
  • Se integra con Maven, Gradle, GitHub Actions, Jenkins y otras herramientas del ecosistema.
  • Las reglas son configurables y pueden extenderse con perfiles de calidad personalizados.

Limitaciones de SonarQube:

  • No puede detectar defectos de lógica de negocio: si la fórmula del precio es incorrecta funcionalmente, SonarQube no lo sabrá. Para eso siguen siendo necesarias las pruebas dinámicas.
  • Genera falsos positivos ocasionales que requieren que el equipo los marque como «Won't Fix» o «False Positive», añadiendo ruido al dashboard si no se gestiona.
  • La versión Community no incluye análisis de ramas (branch analysis), lo que limita su uso en proyectos con múltiples ramas activas sin actualizar a la versión Developer.
  • El análisis de lenguajes como JavaScript o Python es menos preciso que el de Java o C#, donde el tipado estático permite un análisis de flujo de datos más exacto.
Reflexión final: El análisis estático no reemplaza las pruebas unitarias, las pruebas de integración ni la revisión de código por pares. Es una capa complementaria que automatiza la detección de patrones problemáticos conocidos, liberando tiempo de revisión humana para cuestiones de diseño y lógica que las herramientas no pueden evaluar. En un proceso de V&V maduro, ambos enfoques —estático y dinámico— coexisten y se refuerzan mutuamente.
