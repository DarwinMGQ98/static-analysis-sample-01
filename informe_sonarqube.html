1. Introducción

1.1 ¿Qué es el análisis estático de código?

El análisis estático es una técnica que permite revisar el código fuente de un programa sin necesidad de ejecutarlo. La idea es que una herramienta lea el código y busque patrones que se sabe que son problemáticos: cosas como usar un valor que podría ser nulo, tener números sin nombre en el código, o métodos que no dejan claro qué hacen.

La diferencia con las pruebas normales es que aquí no hace falta preparar casos de prueba ni correr el programa. La herramienta analiza el texto del código directamente, lo cual es bastante útil para detectar errores que son difíciles de reproducir con pruebas tradicionales.

1.2 Por qué es importante en la verificación de software

Dentro del proceso de Verificación y Validación, el análisis estático ayuda a responder la pregunta de si estamos construyendo el software de la manera correcta. Su principal ventaja es que detecta problemas muy temprano, antes de que lleguen a producción, donde corregirlos es mucho más costoso y complicado.

Otra ventaja que me pareció importante es que el análisis es completamente automático y repetible. No depende de si el programador tuvo un buen día o no — la herramienta siempre revisa las mismas reglas de la misma manera.

1.3 SonarQube

SonarQube es una plataforma gratuita (en su versión Community) que permite hacer análisis estático de código en varios lenguajes, entre ellos Java. Se instala como un servidor local o remoto y recibe los resultados del análisis que se ejecuta desde Maven o Gradle. Organiza los problemas encontrados en tres categorías:

CategoríaQué significaEjemplos
Bug Código que probablemente cause un error en tiempo de ejecución Posibles NullPointerException, comparaciones incorrectas
Vulnerability Problemas de seguridad que un atacante podría aprovechar Datos sensibles expuestos, algoritmos débiles
Code Smell Código que no es un error pero es difícil de entender o mantener Números mágicos, nombres de métodos sin sentido, métodos muy largos

Además de detectar los problemas, SonarQube indica la severidad de cada uno (Blocker, Critical, Major, Minor, Info) y sugiere cómo corregirlos, lo que facilita saber por dónde empezar.

4. Correcciones Realizadas

Corrección 1 — Bug Critical: posible NullPointerException en el parámetro drink
CampoDetalle
IssueB1
TipoBug
SeveridadCRITICAL
Archivosrc/main/java/ec/edu/epn/Pub.java, línea 12
Commitfix: agregar validacion de nulo en parametro drink para prevenir NullPointerException

¿Cuál era el problema?

El método computeCost recibe un parámetro drink de tipo String y enseguida llama a drink.equals(...). Si alguien llama al método pasando null como bebida, Java lanza una NullPointerException en esa línea porque no se puede llamar a un método sobre un objeto nulo.

SonarQube detecta esto porque analiza todos los posibles valores que puede tener el parámetro y advierte cuando un valor podría ser nulo antes de usarlo.

Código antes

ANTES 
public int computeCost(String drink, boolean student, int amount) {

    // No se verifica si drink es null antes de usarlo
    if (amount > 2 && (drink.equals(GT) || drink.equals(BACARDI_SPECIAL))) {
        throw new RuntimeException("Too many drinks, max 2.");
    }

Código después

DESPUÉS 
public int computeCost(String drink, boolean student, int amount) {
    if (drink == null) {
        throw new RuntimeException("Drink cannot be null");
    }
    if (amount > MAX_COCKTAIL_AMOUNT && (drink.equals(GT) || drink.equals(BACARDI_SPECIAL))) {
        throw new RuntimeException("Too many drinks, max 2.");
    }

Al agregar la validación al inicio del método, si alguien pasa null recibe un mensaje claro en lugar de un error confuso de Java.

Corrección 2 — Code Smell Major: números mágicos sin nombre
CampoDetalle
IssuesCS7 a CS12
TipoCode Smell
SeveridadMAJOR
Archivosrc/main/java/ec/edu/epn/Pub.java
Commitfix: reemplazar numeros magicos por constantes nombradas

¿Cuál era el problema?

En el código original, los precios y valores de configuración estaban escritos directamente como números: 74, 103, 110, 2, 10, etc. Esto se conoce como "números mágicos" porque cualquier persona que lea el código no sabe qué significa ese número sin leer el contexto completo.

Por ejemplo, ¿por qué se divide entre 10? ¿Es un descuento del 10%? ¿Es una tasa de impuesto? Sin un nombre, no queda claro. Y si el descuento cambia, habría que buscar ese 10 en todo el archivo y esperar no cambiar algún otro número que sea coincidencia.

Código antes

ANTES 
// Los números no tienen nombre, no está claro qué significan
if (amount > 2 && (drink.equals(GT) || drink.equals(BACARDI_SPECIAL))) {
    price = 74;
    price = price - price/10;

Código después

DESPUÉS 
// Constantes con nombre al inicio de la clase
private static final int BEER_PRICE             = 74;
private static final int CIDER_PRICE            = 103;
private static final int PROPER_CIDER_PRICE     = 110;
private static final int MAX_COCKTAIL_AMOUNT    = 2;
private static final int STUDENT_DISCOUNT_DIVISOR = 10;

// Ahora en el método el código se entiende solo
if (amount > MAX_COCKTAIL_AMOUNT && ...) {
    price = BEER_PRICE;
    price = price - price / STUDENT_DISCOUNT_DIVISOR;

¿Cuál era el problema?

El código original tenía seis métodos privados llamados ingredient1(), ingredient2(), ..., ingredient6(). Cada uno simplemente devolvía un número. El problema es que con esos nombres no se sabe qué ingrediente representa cada uno sin ir a leer el comentario que tenía al lado.

Esto viola el principio de Clean Code que dice que el código debería explicarse solo, sin depender de comentarios. Si alguien borra los comentarios (o si se está leyendo el código rápidamente), no hay manera de saber qué es ingredient3().

Adicionalmente, el bloque else if de la condición para A_PROPER_CIDER no tenía llaves, lo que también fue reportado como Code Smell por SonarQube.

Código antes

ANTES 
//one unit of rum
private int ingredient1() { return 65; }
//one unit of grenadine
private int ingredient2() { return 10; }
//one unit of lime juice
private int ingredient3() { return 10; }
//one unit of green stuff
private int ingredient4() { return 10; }
//one unit of tonic water
private int ingredient5() { return 20; }
//one unit of gin
private int ingredient6() { return 85; }

// Bloque sin llaves
else if (drink.equals(A_PROPER_CIDER)) price = 110;

Código después

DESPUÉS

4.1 Código final completo — Pub.java

package ec.edu.epn;

public class Pub {
    public static final String ONE_BEER        = "hansa";
    public static final String ONE_CIDER       = "grans";
    public static final String A_PROPER_CIDER  = "strongbow";
    public static final String GT              = "gt";
    public static final String BACARDI_SPECIAL = "bacardi_special";

    private static final int BEER_PRICE              = 74;
    private static final int CIDER_PRICE             = 103;
    private static final int PROPER_CIDER_PRICE      = 110;
    private static final int MAX_COCKTAIL_AMOUNT     = 2;
    private static final int STUDENT_DISCOUNT_DIVISOR = 10;
    private static final int RUM_PRICE               = 65;
    private static final int GRENADINE_PRICE         = 10;
    private static final int LIME_JUICE_PRICE        = 10;
    private static final int GREEN_STUFF_PRICE       = 10;
    private static final int TONIC_WATER_PRICE       = 20;
    private static final int GIN_PRICE               = 85;

    public int computeCost(String drink, boolean student, int amount) {
        if (drink == null) {
            throw new RuntimeException("Drink cannot be null");
        }
        if (amount > MAX_COCKTAIL_AMOUNT
                && (drink.equals(GT) || drink.equals(BACARDI_SPECIAL))) {
            throw new RuntimeException("Too many drinks, max 2.");
        }
        int price;
        if (drink.equals(ONE_BEER)) {
            price = BEER_PRICE;
        } else if (drink.equals(ONE_CIDER)) {
            price = CIDER_PRICE;
        } else if (drink.equals(A_PROPER_CIDER)) {
            price = PROPER_CIDER_PRICE;
        } else if (drink.equals(GT)) {
            price = GIN_PRICE + TONIC_WATER_PRICE + GREEN_STUFF_PRICE;
        } else if (drink.equals(BACARDI_SPECIAL)) {
            price = GIN_PRICE / 2 + RUM_PRICE + GRENADINE_PRICE + LIME_JUICE_PRICE;
        } else {
            throw new RuntimeException("No such drink exists");
        }
        if (student && (drink.equals(ONE_CIDER) || drink.equals(ONE_BEER)
                || drink.equals(A_PROPER_CIDER))) {
            price = price - price / STUDENT_DISCOUNT_DIVISOR;
        }
        return price * amount;
    }
}

6. Conclusiones

Esta tarea me pareció bastante útil porque no es lo mismo leer sobre análisis estático en la teoría que ver en la práctica lo que detecta una herramienta como SonarQube. Al ejecutar el análisis por primera vez y ver que encontró 14 problemas en un archivo relativamente pequeño, quedó claro por qué este tipo de revisión no debería ser opcional en un proyecto real.

Lo que más me llamó la atención fue el bug del NullPointerException. El código compilaba perfectamente y las pruebas unitarias pasaban, pero SonarQube identificó que había un camino en el que el programa podía fallar. Esto demuestra que las pruebas no son suficientes por sí solas para garantizar que el software está bien construido — el análisis estático cubre casos que son difíciles de probar dinámicamente.

También fue interesante aplicar los principios de Clean Code que se vieron en clase. Cosas como renombrar los métodos ingredient1() a constantes con nombres descriptivos hacen que el código sea mucho más fácil de leer. Antes había que ir a buscar el comentario de cada método para entender qué hacía; ahora el código se explica solo. Lo mismo con los números mágicos: un BEER_PRICE dice exactamente qué es ese valor, mientras que un 74 suelto en el código no dice nada.

En cuanto a las ventajas y limitaciones de SonarQube, creo que la mayor ventaja es que es automático y consistente. No se cansa, no omite revisiones por falta de tiempo y aplica las mismas reglas siempre. Como limitación, hay que tener en cuenta que no puede detectar errores de lógica de negocio — si el precio de la cerveza estuviera mal calculado, SonarQube no lo sabría. Para eso siguen siendo necesarias las pruebas unitarias y la revisión humana del código.

En proyectos futuros definitivamente integraría SonarQube desde el inicio, no al final. Si se acumula deuda técnica durante meses, corregirla de golpe es mucho más difícil que ir resolviéndola commit a commit.
